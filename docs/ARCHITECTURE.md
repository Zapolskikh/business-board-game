# Архитектура и точки расширения

Документ объясняет, как устроен проект и как его расширять. Правила игры берутся
из [satirical_business_game_design.md](../satirical_business_game_design.md).

## Слои

```text
┌────────────────────────────────────────────────────────────┐
│ frontend/ (React + Vite)   — только отправляет действия,     │
│                              рисует пришедшее состояние      │
└───────────────▲────────────────────────────────────────────┘
                │ REST (/api)
┌───────────────┴────────────────────────────────────────────┐
│ backend/app/ (FastAPI)     — тонкий слой: маршруты + хранилище│
├─────────────────────────────────────────────────────────────┤
│ backend/simulation/        — боты + прогон партий + статистика│
├─────────────────────────────────────────────────────────────┤
│ backend/game_engine/       — ЧИСТАЯ логика игры (без веба)   │
│   data/ (JSON)             — роли, клетки, поля, баланс       │
└─────────────────────────────────────────────────────────────┘
```

Ключевая инвариант: **`game_engine` не знает о FastAPI и симуляции**. Он
запускается в тестах и в симуляции без сети. Это делает правила
тестируемыми и симулируемыми.

## Поток «действие → состояние»

Движок — маленький автомат на двух действиях: `roll_dice` и `resolve_decision`.

```text
roll_dice
  → бросок кубика, движение (бонус Старта, возможное повышение круга)
  → активация клетки, на которую попал игрок
       → клетка либо завершает эффект сразу,
       → либо выставляет state.pending_decision (нужен выбор игрока)
resolve_decision (повторяется, пока есть pending_decision)
  → ответ маршрутизируется обратно в клетку, поднявшую решение
→ доп. бросок (если есть) ИЛИ переход хода к следующему игроку
→ проверка победы / лимита раундов
```

И человек в браузере, и бот в симуляции отвечают на решение **одинаково** —
действием `resolve_decision` с `option_id`. Поэтому один и тот же код обслуживает
живую игру и симуляцию.

## Точка расширения 1 — новая клетка

Клетки регистрируются по строковому ключу через реестр
([registry.py](../backend/game_engine/registry.py)). Ядро не содержит `if type == ...`.

1. Реализуйте поведение (стейтлес-класс) в `game_engine/cells/*.py`:

   ```python
   from game_engine.cells.base import BaseCell
   from game_engine.registry import register_cell

   @register_cell("mall")  # Торговый центр
   class MallCell(BaseCell):
       def on_land(self, engine, player, cell):
           if cell.owner_id is None:
               # поднять решение о покупке (см. cells/common.py: buy_options)
               ...
           elif cell.owner_id != player.id:
               owner = engine.state.player_by_id(cell.owner_id)
               rent = engine.balance.ring_value("prices.mall", cell.ring) // 5
               engine.transfer_money(player, owner, rent, reason="аренда ТЦ")
   ```

2. Импортируйте модуль в [cells/\_\_init\_\_.py](../backend/game_engine/cells/__init__.py).
3. Добавьте метаданные типа в [data/cells.json](../backend/data/cells.json)
   (title, buyable, price_key, tags, color).
4. Добавьте клетку в раскладку [data/boards/board_72.json](../backend/data/boards/board_72.json).

Движок, боты и UI подхватят её автоматически. Неизвестный тип клетки не роняет
игру — используется безопасный `__fallback__` (no-op).

### Правила для клеток

- Клетки **не мутируют состояние напрямую**, а вызывают хелперы движка:
  `grant_money`, `charge_money`, `transfer_money`, `add_scandal`, `add_roof`,
  `consume_roof`, `set_role`, `remove_role`, `send_to_hospital`, `send_to_jail`,
  `grant_experience`, `lose_experience`. Так сквозные правила (2 скандала →
  потеря роли, банкротство, Крыша) живут в одном месте и всегда согласованы.
- Эффект **владения не зависит от роли** владельца (принцип из дизайна).
- Ролевой эффект применяется **только в момент попадания** на клетку.

## Точка расширения 2 — новая роль

1. Добавьте роль в [data/roles.json](../backend/data/roles.json)
   (id, title, color, themes, weaknesses).
2. Добавьте id в перечисление `Role` в
   [enums.py](../backend/game_engine/enums.py) (ветвления клеток идут по нему).
3. В нужных клетках обработайте `self.has_role(player, Role.NEW)`.

## Точка расширения 3 — размер и состав поля

Полностью данные: [data/boards/board_72.json](../backend/data/boards/board_72.json).

- `ring_sizes` — число клеток в каждом круге.
- `distribution[ring]` — сколько клеток каждого типа в круге. Сумма не обязана
  совпадать с `ring_sizes`: генератор добивает недостачу типом `filler_type` и
  срезает избыток. Поэтому «поменять размер поля» = поменять одно число.
- `fixed[ring][slot]` — жёстко закреплённые клетки (например, Старт на слоте 0).

Новый вариант поля — новый файл `data/boards/board_XX.json`; запуск:
`python -m simulation.cli --board board_XX`.

## Точка расширения 4 — баланс

Все числа — в [data/balance.json](../backend/data/balance.json): бонусы
Старта/банка по кругам (массивы `[круг1, круг2, круг3]`), цены объектов, штрафы,
множители казино, цена Крыши, условия победы, правила повышения круга.

Значения — либо скаляр, либо массив по кругам; читаются через
`balance.ring_value("path", ring)`. Меняйте JSON и прогоняйте симуляцию.

## Симуляция и поиск дисбаланса

```bash
python -m simulation.cli --games 1000 --board board_72 --seed 42
```

- Боты (`RandomBot`, `GreedyBot`) отвечают на решения; кубики — автоматически.
- Каждая партия воспроизводима по seed; RNG решений бота отделён от RNG игры,
  чтобы изменение набора решений не сдвигало поток кубиков.
- Отчёт: **win-rate по стартовой роли** (главный сигнал баланса — при случайных
  ботах роли должны иметь близкий win-rate ≈ 1/N), средний капитал, банкротства,
  частота попаданий на клетки.

Пример из текущего прототипа (числа — заглушки, требуют балансировки):
Капиталист ~29% vs Мафиози ~20% win-rate → есть перекос, который и нужно
устранять правкой `balance.json`.

- **Карты «?»**: маленькая взвешенная случайность вместо колоды карт.
- **Банкротство**: откат (потеря дешёвого объекта + опыт), не вылет — как в
  дизайне, но точные правила ещё предстоит определить.
- **Мультиплеер по сети** (lobby/rooms/WebSocket) — Milestone 3, ещё не сделан;
  сейчас хранилище игр в памяти ([app/game_manager.py](../backend/app/game_manager.py)).

Все пункты локализованы и расширяются без переписывания ядра.
