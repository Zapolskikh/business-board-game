# План production-релиза «Города влияния»

Этот файл — канонический план переноса текущей браузерной игры в production-архитектуру. Он нужен, чтобы работу можно было продолжить после прерывания любой сессии без восстановления договорённостей из чата.

## Зафиксированные решения

- Ветка разработки: `production-game-engine`.
- Клиент: React/Vite. Клиент рисует игру, а не определяет результат действий.
- Авторитетный backend: Python + FastAPI.
- Транспорт первого релиза: REST и адаптивный polling. WebSocket пока не нужен.
- Все правила живут в независимом Python-пакете `city_engine` без зависимостей от FastAPI, React и хранилища.
- Живая партия, серверные боты и массовые симуляции используют один и тот же `city_engine`.
- Первое production-хранилище: существующий Upstash Redis из Vercel (`upstash-kv-blue-tree`).
- Доступ без регистрации: игрок видит активные комнаты, вводит пароль и выбирает свободное человеческое место. Постоянная идентичность и восстановление занятого места пока не требуются.
- Комната хранится после отключения игроков. Ожидающие и завершённые комнаты позднее получат отдельную политику очистки.
- Архитектура хранилища строится через `RoomRepository`, чтобы позднее заменить Upstash на PostgreSQL без изменения движка, ботов, API-команд и React.
- Старая игра с кубиком и клеточным полем удалена из репозитория. Новый `backend/city_engine` является единственным игровым движком проекта.

## Целевая архитектура

```text
React
  -> REST command
FastAPI / CityGameService
  -> load room
  -> validate actor and revision
  -> city_engine.apply(command)
  -> run server bots through the same engine
  -> compare-and-save room
UpstashRoomRepository

city_engine <- live API
city_engine <- bot policies
city_engine <- simulations
```

## Доменная модель

`city_engine` принимает `GameState` и одну типизированную команду и возвращает новое состояние вместе с доменными событиями.

Основные команды:

- `buy_asset`, `sell_asset`, `improve_asset`, `buy_capacity`;
- `buy_action_card`, `play_action_card`, `convert_action_card`;
- `claim_role`, `use_role_power`;
- `develop_district`, `basic_action`, `city_project`, `buy_roof`;
- `end_turn`.

Обязательные свойства движка:

- детерминированность при одинаковом seed и последовательности команд;
- JSON-сериализация без `pickle`;
- `schema_version`, `rules_version`, `content_version` в состоянии;
- серверная проверка права хода, ресурсов, целей и допустимости команды;
- отдельные RNG игры и политики бота;
- `legal_actions(state, player_id)` как единственный источник допустимых действий для людей и ботов;
- проекция `view_for(state, viewer_id)`, скрывающая руки соперников и порядок колод;
- журнал доменных событий для анимации, диагностики и будущего replay.

## Канонический контент

Роли, районы, объекты, карты действий и события перенесены из прежнего `frontend/src/city/data.ts` в версионируемый backend JSON. Frontend получает их через `GET /api/city/meta`; TypeScript больше не является источником игровых правил.

До завершения переноса необходимо явно зафиксировать спорные правила, найденные аудитом:

- развитие района персональное или глобальное;
- одно событие на всю игру;
- подготовка первого игрока нового раунда выполняется ровно один раз;
- актуальные версии дани Мафиози, дохода Журналиста, санкции Силовика и цены переворота;
- правила редкостей и ротации рынка.

## Комнаты и lobby

Состояния комнаты: `waiting`, `playing`, `finished`.

Поток первого релиза:

1. Хост создаёт комнату: название, пароль, настройки и состав мест.
2. Комната появляется в списке активных комнат.
3. Гость открывает комнату, вводит пароль и выбирает свободное человеческое место.
4. Хост может добавлять/убирать ботов и запускает игру.
5. После отключения игрок снова вводит пароль и выбирает любое доступное место. Постоянное закрепление личности пока отсутствует намеренно.

Пароль хранится только как `scrypt`/Argon2-хеш. Знание пароля разрешает вход в lobby, но каждая команда дополнительно содержит выбранный `player_id`; сервер проверяет, что место активно, является человеческим и сейчас имеет право действовать. Более строгие seat-токены добавляются позднее при появлении регистрации.

## REST API первого релиза

```text
GET    /api/city/meta
GET    /api/city/rooms
POST   /api/city/rooms
GET    /api/city/rooms/{room_id}
POST   /api/city/rooms/{room_id}/join
POST   /api/city/rooms/{room_id}/seats
POST   /api/city/rooms/{room_id}/start
GET    /api/city/rooms/{room_id}/state?after_revision=N
POST   /api/city/rooms/{room_id}/commands
```

Каждая мутирующая команда получает `command_id` и `expected_revision`. Несовпадение revision возвращает `409 Conflict`. Повторный `command_id` не выполняется второй раз.

Polling:

- 3–5 секунд во время lobby и чужого хода;
- 15–30 секунд в скрытой вкладке;
- после собственной команды используется её ответ, отдельный polling не нужен;
- после завершения игры polling останавливается;
- сетевые ошибки включают exponential backoff.

## Upstash Redis

Новый код не использует существующую схему `одна комната + pickle + TTL 6 часов`.

Минимальные ключи:

```text
city:room:{room_id}        # JSON комнаты и игры
city:rooms:active          # индекс активных комнат
```

Сохранение выполняется атомарным compare-and-set по `revision` через Lua/EVAL. Движок ничего не знает о Redis.

Политика хранения первого релиза:

- пустая ожидающая комната: ограниченный TTL;
- начатая партия: без короткого TTL;
- завершённая партия: длительный TTL;
- автоматическая очистка и архивирование уточняются после появления реального трафика.

## Боты и симуляции

Бот выбирает только из результата `legal_actions` и возвращает обычную команду. Команда проходит через `city_engine.apply` так же, как команда человека.

После команды человека backend выполняет действия ботов до следующего человеческого решения и сохраняет комнату одной атомарной записью. API возвращает последовательность событий; React проигрывает их как анимацию без задержек внутри backend.

Старый `frontend/scripts/simulate-city.mjs` удалён после того, как Python-runner начал воспроизводить полную production-партию. Сохранённые старые отчёты остаются историческими и не используются для финальной балансировки.

## Критерии готовности первого релиза

- На сайте одновременно существует несколько независимых комнат.
- Комната переживает отключение всех браузеров и serverless cold start.
- Гость находит комнату, вводит пароль, выбирает свободное место и продолжает игру.
- Две одновременные команды не могут затереть друг друга.
- Клиент не может выполнить нелегальное действие изменением HTTP payload.
- Серверные боты и балансный runner используют тот же движок, что и люди.
- Любая партия воспроизводится по seed и журналу команд.
- Ни одна карта соперника или закрытая колода не раскрывается чужому клиенту.
- Смена Upstash на PostgreSQL требует только нового `RoomRepository` и мигратора данных.

## Не входит в первый релиз

- регистрация и аккаунты;
- постоянное владение местом после отключения;
- рейтинги и matchmaking;
- WebSocket;
- spectator permissions и moderation;
- платёжная система;
- мобильные приложения;
- горизонтальное масштабирование нескольких backend-инстансов через event bus.
